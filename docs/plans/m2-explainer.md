M2 builds a provider-agnostic local HTTP proxy that sits between AI coding agents and their upstream API providers. When an agent sends a request through the proxy, the system normalizes the payload into a canonical schema via declarative mapping specs, classifies any tool intents using the central taxonomy from M1.5, evaluates commandments against the normalized event, and either blocks the request before it ever leaves the machine or forwards it upstream with optional egress redaction. On the response side, it extracts token usage, cost, finish reason, and error metadata from both streaming (SSE) and non-streaming responses, producing a single audited event per request that flows through the existing hash-chain audit pipeline. OpenAI-compatible transport ships as the first implementation, but the transport interface, mapping spec format, and routing layer are designed so that adding Anthropic or Google requires only a new YAML mapping file and a transport registration — no changes to the enforcement or audit core.
This is the third vertical slice in the prototype roadmap laid out in the implementation design doc. M0 delivered the daemon, log watcher, audit trail, and CLI foundation. M1/M1.5 added the commandment engine with warn-only enforcement and a central classifier that normalizes tool taxonomy across Claude Code and Codex. M2 is where enforcement becomes preventative — the proxy is the first adapter that can actually block a request, not just observe and warn after the fact. It also closes the telemetry loop by capturing cost and usage data directly from provider responses rather than parsing it from third-party log files. Together these three milestones give M3 everything it needs: the TUI gets real-time governance events with cost counters to render, the OTel exporter gets complete GenAI spans with reliable usage dimensions, and the release hardening phase inherits already-verified streaming correctness and enforcement behavior.

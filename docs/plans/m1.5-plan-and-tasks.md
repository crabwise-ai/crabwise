M1.5 — Canonical Tool Taxonomy + Central Classifier

   Goal: Replace per-adapter bespoke tool classification with a single, data-driven, provider-aware taxonomy so
   commandments rules are stable across providers and ready for M2 block enforcement.

   Demo: Same commandment (tool_category: shell, tool_effect: execute) catches Bash from Claude Code,
   run_shell_command from Codex, and a future proxy-intercepted exec_command -- without any rule changes.

   Implementation stance: provider-aware taxonomy now, with deterministic classification behavior and
   explicit provenance so M2 block enforcement can trust matches.

   P0 hardening requirements (must-have):
   - [x] Deterministic arg key extraction contract (nested object flattening, lowercase normalization,
     stable ordering, array handling rules) shared by all adapters.
   - [x] Atomic dual reload contract: on SIGHUP, parse+compile commandments and tool registry first,
     then perform one swap; if either fails, keep both previous versions.
   - [x] ActionType derivation boundary: derive only for tool-origin events; never overwrite ai_request,
     system, or unknown event action types.
   - [x] Classification provenance on every classified tool event (`classification_source` = exact|pattern|heuristic|fallback).
   - [ ] Cross-adapter conformance matrix: equivalent Claude/Codex/proxy tool actions must classify identically.
   - [x] Deterministic exact lookup order across provider and `_default` fallback.
   - [x] Runtime visibility for taxonomy drift via `unclassified_tool_count` in daemon status.

   ──────────────────────────────────────────

   What Changes

   1. New `internal/classify` package

   Central classifier with three layers, evaluated in order:

     1. Exact match: (provider, tool_name) -> category + effect
     2. Pattern match: tool_name matches glob -> category + effect
     3. Heuristic: arg key names (command, path, url) -> category + effect
     4. Fallback: category=other, effect=unknown

   Exact lookup order (deterministic):
   1. providers[provider].tools[tool_name] (case-sensitive)
   2. providers[provider].tools[lower(tool_name)]
   3. providers["_default"].tools[lower(tool_name)]

   Notes:
   - Regex matching is intentionally deferred from v1 classifier rules to reduce complexity/risk.
   - Classifier returns provenance (`classification_source`) alongside category/effect.
   - Heuristic rules are evaluated top-to-bottom; first match wins.
   - Registry loader should emit a warning when heuristic globs have high-overlap potential.

   Taxonomy (v1):

   go
     // tool_category (coarse intent)
     const (
         CategoryShell       = "shell"          // bash, exec, command
         CategoryFileRead    = "file.read"      // read, cat, open
         CategoryFileWrite   = "file.write"     // write, save, create
         CategoryFileEdit    = "file.edit"      // edit, patch, replace
         CategoryFileSearch  = "file.search"    // grep, glob, find, list
         CategoryNetwork     = "network"        // http, fetch, curl
         CategoryCodeExec    = "code.exec"      // python eval, js eval
         CategoryBrowser     = "browser"        // navigate, screenshot
         CategoryOther       = "other"          // unclassified
     )

     // tool_effect (side-effect level)
     const (
         EffectReadOnly     = "read_only"
         EffectMetadata     = "metadata"       // list files, stat
         EffectMutation     = "mutation"       // write, edit, create
         EffectExecute      = "execute"        // runs arbitrary code/commands
         EffectUnknown      = "unknown"        // couldn't determine
     )

   2. YAML tool registry (`configs/tool_registry.yaml`)

   Provider-aware, hot-reloadable via SIGHUP (same mechanism as commandments). Versioned.

```

     providers:
       anthropic:
         tools:
           Bash:
             category: shell
             effect: execute
           Read:
             category: file.read
             effect: read_only
           Write:
             category: file.write
             effect: mutation
           Edit:
             category: file.edit
             effect: mutation
           Glob:
             category: file.search
             effect: metadata
           Grep:
             category: file.search
             effect: read_only

       openai:
         tools:
           run_shell_command:
             category: shell
             effect: execute
           exec_command:
             category: shell
             effect: execute
           execute_command:
             category: shell
             effect: execute
           read_file:
             category: file.read
             effect: read_only
           write_file:
             category: file.write
             effect: mutation
           edit_file:
             category: file.edit
             effect: mutation
           list_files:
             category: file.search
             effect: metadata
           search_file:
             category: file.search
             effect: read_only

       _default:  # fallback for unknown providers
         tools:
           bash:
             category: shell
             effect: execute
           shell:
             category: shell
             effect: execute

     heuristics:
       - match:
           name_glob: ["*bash*", "*shell*", "*exec*", "*command*"]
         set:
           category: shell
           effect: execute

       - match:
           name_glob: ["*read*", "*cat*", "*open*"]
           arg_keys_any: [path, filename, file_path]
         set:
           category: file.read
           effect: read_only

       - match:
           name_glob: ["*write*", "*save*", "*create*"]
           arg_keys_any: [path, filename, content, data]
         set:
           category: file.write
           effect: mutation

       - match:
           name_glob: ["*edit*", "*patch*", "*replace*"]
         set:
           category: file.edit
           effect: mutation

       - match:
           name_glob: ["*grep*", "*glob*", "*find*", "*search*", "*list*"]
         set:
           category: file.search
           effect: metadata

       - match:
           name_glob: ["*http*", "*fetch*", "*curl*", "*request*"]
         set:
           category: network
           effect: execute

       - match:
           arg_keys_any: [command, cmd]
         set:
           category: shell
           effect: execute

       - match:
           arg_keys_any: [url, endpoint, uri]
         set:
           category: network
           effect: execute
```

   3. AuditEvent schema additions

   New fields on AuditEvent:

   go
     ToolCategory     string `json:"tool_category,omitempty"`     // "shell", "file.read", etc.
     ToolEffect       string `json:"tool_effect,omitempty"`       // "execute", "read_only", etc.
     ToolName         string `json:"tool_name,omitempty"`         // raw tool name preserved
     TaxonomyVersion  string `json:"taxonomy_version,omitempty"`  // "v1"
     ClassificationSource string `json:"classification_source,omitempty"` // exact|pattern|heuristic|fallback

   •  ActionType remains, derived from ToolCategory only for tool-origin events (shell -> command_execution,
      file.* -> file_access, etc.)
   •  New fields included in hash chain (appended to CanonicalBytes)
   •  Event schema is updated directly for fresh DB bootstrap (no incremental migration requirement)
   •  New fields indexed: idx_events_tool_category, idx_events_tool_effect

   4. Adapter changes

   Both parsers (claudecode.go, codexcli.go) stop calling their local classifyTool() / classifyCodexTool().
   Instead:

   1. Set ToolName to the raw tool name
   2. Set Provider (already "anthropic" / "openai" from adapter context, or inferred)
   3. Build argKeys via shared deterministic extractor (not adapter-specific ad hoc parsing)
   4. After event construction, call classify.Classify(provider, toolName, argKeys) to populate ToolCategory,
      ToolEffect, TaxonomyVersion, ClassificationSource
   5. Derive ActionType from ToolCategory for backward compat (tool-origin events only)

   The classifyTool() and classifyCodexTool() functions are deleted.

   5. Commandments engine: new matchable fields

   eventFieldValue() gains three new fields: tool_category, tool_effect, tool_name. Rules can now match on them:

   yaml
     # Before (fragile -- misses codex "run_shell_command"):
     - name: no-destructive-commands
       match:
         action_type: command_execution
         arguments:
           type: regex
           pattern: "rm\\s+-rf"

     # After (provider-agnostic):
     - name: no-destructive-commands
       match:
         tool_category: shell
         tool_effect: execute
         arguments:
           type: regex
           pattern: "rm\\s+-rf"

   Both styles work simultaneously. action_type matching still works (backward compat).

   6. Default commandments update

   Update configs/commandments_default.yaml to use tool_category where appropriate, keeping action_type matches
   for non-tool events (ai_request, system).

   7. SIGHUP reload for tool registry

   Extend daemon's SIGHUP handler to reload both commandments and tool registry atomically. Emit
   tool_registry_reload_ok / tool_registry_reload_failed system events.

   Daemon observability:
   - Track classifier fallthroughs (`category=other`, `effect=unknown`) and expose `unclassified_tool_count`
     via daemon status IPC and `crabwise status`.

   Atomic reload behavior:
   1. Parse+compile new commandments and registry into candidate structures
   2. If either fails, emit *_reload_failed and keep active versions unchanged
   3. If both succeed, swap both in one critical section and emit *_reload_ok

   8. CLI classifier introspection

   Add `crabwise classify <tool-name> [--provider <p>] [--args key1,key2,...]` to dry-run classifier
   resolution and print category/effect/source/taxonomy version.

   What This Enables for M2

   When the proxy intercepts a live request in M2:
   1. It has provider from the upstream URL
   2. It has tool_name from the request payload
   3. It has arg_keys from the arguments
   4. It calls the same classify.Classify() -- same taxonomy, same rules
   5. Block enforcement works identically whether the event came from a log file or a live proxy intercept

   No adapter-specific classification code to duplicate.
